## 1. New Message Type

### `control_surface_state`

- **Sender:** Desktop
- **Receiver:** Mobile
- **Purpose:** Projects the active Profile's control surface (Profile → Pages
  → Buttons hierarchy) to connected clients.
- **Direction:** Desktop-initiated (push). Only sent to trusted clients.
- **v0.5:** Proposed.

### Payload contract (v1)

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | yes | Always `"control_surface_state"` |
| `schema_version` | integer | yes | The exact integer `1` for v1 |
| `profile_id` | string or null | yes | Profile ID this surface belongs to; null if no Profile active |
| `profile_name` | string or null | yes | Profile display name; null if no Profile active |
| `pages` | array or null | yes | PageProjection array; null if no Profile active |

#### PageProjection

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `page_id` | string | yes | Opaque Page identity (timestamp-hex String newtype) |
| `name` | string | yes | Page display name |
| `buttons` | array | yes | ButtonProjection array |

#### ButtonProjection

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `button_id` | string | yes | Opaque Button identity (timestamp-hex String newtype) |
| `label` | string | yes | Button display label |

### Identity and order semantics

**ButtonId:**
- Derived from model.rs `ButtonId(String)` — timestamp-hex String newtype
- Uniqueness is enforced per-page only (command.rs AddButton, MoveButton
  checks), not cross-page or cross-Profile
- The projection communicates ButtonId value as-is; Mobile treats it as
  an opaque stable identity
- The projection does NOT rely on Document-global ButtonId uniqueness

**PageId:**
- Derived from model.rs `PageId(String)` — timestamp-hex String newtype
- No uniqueness enforcement in the authoritative model
- The projection communicates PageId value as-is for stable reference

**Order:**
- Page array order mirrors authoritative Vec<Page> index
- Button array order mirrors authoritative Vec<Button> index per Page

### Nullability invariants

| profile_id | profile_name | pages | Meaning |
|------------|-------------|-------|---------|
| null | null | null | No active Profile |
| string | string | array | Active Profile with surface |

If profile_id is null:
  → profile_name MUST be null
  → pages MUST be null

If profile_id is string:
  → profile_name MUST be string
  → pages MUST be array

Any other combination is structurally invalid.

`pages: null` means no active Profile. `pages: []` means active Profile with
zero Pages. These are architecturally distinct.

### Unresolved active Profile

If the Desktop cannot resolve the active Profile identity against the
authoritative Document/configuration, the derivation fails. No
`control_surface_state` frame is published. The CSS channel is unchanged.
This condition is logged.

### Association and eligibility

Mobile independently retains:
- latest valid `ActiveProfileStateMessage` (APS)
- latest valid `ControlSurfaceStateMessage` (CSS)

A structurally valid CSS frame ALWAYS replaces the retained latest CSS state,
even if the CSS profile_id does not match the APS active_profile_id.

Eligibility is derived from retained state:

| APS active_profile_id | CSS profile_id | CSS eligible? |
|----------------------|----------------|---------------|
| non-null, matches CSS | non-null | yes |
| non-null, differs | non-null | no (retained but ineligible) |
| null | any | no |
| any | null | no |

**Profile with Pages:**
```json
{
  "type": "control_surface_state",
  "schema_version": 1,
  "profile_id": "0000000000000000a1b2c3d4e5f6a7b8",
  "profile_name": "Coding",
  "pages": [
    {
      "page_id": "00000000000000001122334455667788",
      "name": "Main",
      "buttons": [
        { "button_id": "0000000000000000aabbccddeeff0011", "label": "Compile" },
        { "button_id": "00000000000000002233445566778899", "label": "Test" }
      ]
    }
  ]
}
```

**No active Profile:**
```json
{
  "type": "control_surface_state",
  "schema_version": 1,
  "profile_id": null,
  "profile_name": null,
  "pages": null
}
```

**Active Profile with zero Pages:**
```json
{
  "type": "control_surface_state",
  "schema_version": 1,
  "profile_id": "00000000000000001122334455667788",
  "profile_name": "Empty",
  "pages": []
}
```

### Android receiver behaviour

| Scenario | Behaviour |
|----------|-----------|
| Valid CSS received | Parse, retain, update UI/diagnostic |
| Unsupported schema_version | Log, ignore, stay connected |
| Nullability invariant violation | Log, ignore, stay connected |
| Missing required field | Log, ignore, stay connected |
| Malformed JSON | Log, ignore, CSS unchanged |
| profile_id == null | No eligible CSS |
| profile_id mismatch with APS | Retain CSS, mark ineligible |
| profile_id matches APS | CSS eligible as current |

### Unknown fields

Unknown fields in a CSS frame are ignored. Strict parsing applies to
the known v1 field set.

## 2. Compatibility

### `active_profile_state` (v0.4)

- Unchanged, frozen
- No fields added or removed
- Delivery order: no guaranteed order between APS and CSS. Initial trusted
  implementation MAY emit APS then CSS sequentially. Mobile correctness
  MUST NOT depend on this ordering.

### Existing message types

All v0.3/v0.4 message types unchanged.

### Schema versioning

| Message | schema_version | Since |
|---------|---------------|-------|
| `active_profile_state` | 1 | v0.4 |
| `control_surface_state` | 1 | v0.5 |

## 3. Derivation Function

```
(active_profile_id: Option<ProfileId>,
 profiles: &[Profile])
    → DerivationResult<ControlSurfaceState>
```

| Input | Output |
|-------|--------|
| active_profile_id = None | Ok: { profile_id: null, profile_name: null, pages: null } |
| active_profile_id = Some(id) but Profile(id) not found in profiles | DerivationFailure |
| active_profile_id = Some(id), Profile found with zero Pages | Ok: { profile_id: id, profile_name: name, pages: [] } |
| active_profile_id = Some(id), Profile has Pages | Ok: { profile_id: id, profile_name: name, pages: [...Pages with Buttons] } |

DerivationFailure:
- Not a valid no-surface snapshot
- The CSS channel retains its previous value
- The derivation failure is logged

## 4. Field Semantics

### `type`

Message discriminator. Must be exactly `"control_surface_state"`.

### `schema_version`

Required. The v0.5 Android receiver accepts only the exact integer `1`.
A non-integer or unsupported integer causes rejection of the frame.

### `profile_id`

Required. String when an active Profile is resolved, null when no Profile
is active. A missing profile_id is invalid. The value corresponds to the
Desktop's ProfileId as_string() output.

### `profile_name`

Required. String when an active Profile is resolved, null when no Profile
is active. Derived from Profile.name. A profile_name of empty string is
valid for a Profile with no name.

### `pages`

Required. Array of PageProjection when an active Profile is resolved, null
when no Profile is active. Each element preserves the Page's order from the
authoritative Vec<Page>.

### `pages[*].page_id`

Required string. Opaque identity matching the authoritative PageId string.
Mobile treats this as a stable reference — no semantic interpretation.

### `pages[*].name`

Required string. Display name from Page.name.

### `pages[*].buttons`

Required array. ButtonProjection entries preserving Button order from the
authoritative Vec<Button>. May be empty.

### `pages[*].buttons[*].button_id`

Required string. Opaque identity matching the authoritative ButtonId string.
The projection does not rely on Document-global uniqueness. Mobile treats
this as a stable opaque reference.

### `pages[*].buttons[*].label`

Required string. Display label from Button.label. May be empty.

## 5. Cross-Projection Ordering

No atomic identity+surface delivery is required. No cross-projection
ordering guarantee exists. Mobile correctness must not depend on APS arriving
before CSS or vice versa.

Initial trusted-snapshot implementation MAY intentionally send APS before
CSS for deterministic diagnostic display. This is an implementation choice,
not a protocol invariant.

Subsequent projection updates to either channel have no guaranteed
cross-channel ordering.

## 6. Invalidation

| Event | CSS published |
|-------|---------------|
| Active Profile identity changes | Yes |
| Active Profile rename | Yes |
| Page added | Yes |
| Page renamed | Yes |
| Page removed | Yes |
| Button added | Yes |
| Button relabeled | Yes |
| Button reordered | Yes |
| Button removed | Yes |
| Inactive Profile edit | No |
| Context observation, no active Profile change | No |
| Unresolved Profile derivation failure | No (channel unchanged) |

## 7. Trust Model

Same trust gate as v0.4. No new trust state is introduced for CSS.
