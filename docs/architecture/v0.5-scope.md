# v0.5 — Scope

**Status:** Draft
**Date:** 2026-07-15

Normative references: ADR-021, v0.4-scope.md, model.rs.

---

## 1. Goal

Project the active Profile's control surface to paired Mobile clients as a
second independent projection stream, alongside the existing
`active_profile_state`. The projection preserves the certified
Profile → Pages → Buttons hierarchy and projects structural identity and
display presentation only.

## 2. In Scope

| # | Item | Rationale |
|---|------|-----------|
| 1 | `control_surface_state` wire message type | New message carrying control surface snapshot |
| 2 | Profile → Pages → Buttons nested hierarchy | Certified domain model (model.rs) |
| 3 | Profile association (profile_id + profile_name) | Explicit surface-to-Profile binding |
| 4 | ButtonId as opaque stable identity | Mobile references controls by Identity |
| 5 | Full snapshot delivery (not delta) | ADR-015 latest-value semantics |
| 6 | Null triple for no active Profile | profile_id=null, profile_name=null, pages=null |
| 7 | Active Profile with zero Pages: pages=[] | Distinguish from no active Profile |
| 8 | Independent watch channel for CSS | Failure isolation from APS |
| 9 | Derivation: Document → ControlSurfaceState | Compute projection from Profile document |
| 10 | Invalidation on active Profile change | New Profile → new CSS |
| 11 | Invalidation on Page/Button structure edit | Add/rename/remove/reorder → re-project |
| 12 | schema_version = 1 | Self-describing schema |
| 13 | No-op suppression | Derivation unchanged → no publish |
| 14 | Unresolved active Profile = derivation failure | No fabricated empty projection |
| 15 | Android diagnostic spike | parse, validate, retain, derive eligibility, display/log hierarchy |

## 3. Out of Scope

| # | Item | Rationale |
|---|------|-----------|
| 1 | ActionReference in wire format | Desktop domain type — ADR-021 rejects |
| 2 | assigned (or equivalent execution state) | v0.5 is structural identity and display only |
| 3 | enabled, configured, has_action, action_available | Same architectural boundary |
| 4 | Delta/incremental projection | Full snapshot sufficient |
| 5 | Projection revision counter | schema_version suffices |
| 6 | Production deck UI on Mobile | v0.5 is diagnostic spike |
| 7 | Production tappable buttons on Mobile | Diagnostic display only |
| 8 | ButtonId action request from Mobile | Action routing is v0.6+ |
| 9 | Control surface editing from Mobile | v0.5 is projection-only |
| 10 | Control surface persistence on Mobile | In-memory retained state only |
| 11 | Atomic co-delivery of APS + CSS | Independent projections |

## 4. Preserved Contracts

All v0.4 contracts remain unchanged.

## 5. Android Role (v0.5 Spike)

The Android spike validates that a Mobile receiver can:

- discriminate the new CSS message type
- apply a strict v1 parser
- validate schema_version as exact integer 1
- validate nullability invariants (profile_id / profile_name / pages)
- parse PageProjection array
- parse ButtonProjection array per Page
- retain Page order
- retain Button order per Page
- reject invalid updates without corrupting retained state
- independently retain APS and CSS messages
- derive CSS eligibility from retained APS identity + CSS profile_id
- tolerate temporary association mismatch
- converge safely regardless of APS/CSS arrival order

The spike does NOT build:
- Production command deck UI
- Tappable/interactive button widgets
- Execution sender for ButtonId actions
- Action request routing
- Structured action_result consumer
- Compose architecture or navigation framework
- Room persistence
- Production lifecycle redesign

Diagnostic/logging display is acceptable. Example diagnostic output:

```
Profile: Coding [id]
Page: Main [page-id]
  1. Compile [button-id]
  2. Test [button-id]
Eligible: true
```

## 6. Desktop Role

- Derive ControlSurfaceState from active Profile Document
- Serialize to CSS JSON via strict DTO
- Publish via independent watch channel
- Compare new derivation against previous value (no-op if unchanged)
- Handle serialization failure (log, drop, channel unchanged)
- Handle derivation failure (log, drop, channel unchanged — no fabricated surface)
- Deliver to all trusted connections via cloned receivers

## 7. Success Condition

v0.5 is complete when:

1. A paired Mobile client receives `control_surface_state` for the active
   Profile
2. The diagnostic display shows the correct Profile → Pages → Buttons
   hierarchy
3. Page order and Button order match the authoritative Desktop Document
4. No active Profile produces the null triple
5. Active Profile with zero Pages produces pages=[]
6. Document edit (add/rename/remove/reorder Page or Button) triggers
   re-projection
7. Reconnect delivers latest CSS snapshot
8. APS/CSS mismatch retains CSS as ineligible and converges safely
9. The `active_profile_state` v1 message is unchanged
10. No Desktop domain types appear in the wire format
11. No assigned or equivalent execution-derived field exists in the contract
