## 1. Objective

Introduce a Workflow Engine that allows users to define multi-step
automation sequences on the Desktop and execute them from a single
Android invocation. Workflows compose existing atomic actions into
ordered, optionally conditional, reusable sequences.

## 2. In Scope

| # | Item | Rationale |
|---|------|-----------|
| 1 | `Workflow` domain model | Named sequence of steps, persisted in Document |
| 2 | `WorkflowStep` model | Single action invocation with optional parameters |
| 3 | `WorkflowDefinition` in Document | Desktop-authoritative workflow storage |
| 4 | `Workflow` command variants | Create, update, delete, reorder workflows |
| 5 | `WorkflowRegistry` | Resolves workflow_id ‚Üí definition at execution time |
| 6 | `execute_workflow()` | Sequential step executor with stop-on-failure |
| 7 | `workflow_invoke` message type | Mobile triggers a workflow by opaque workflow_id |
| 8 | `workflow_invoke_result` response | Per-step execution report with overall status |
| 9 | Step-level execution history | In-memory ring buffer of recent workflow runs |
| 10 | Android `WorkflowRequest` | Outbound serialization for workflow invocation |
| 11 | Android result parsing | Parse workflow_invoke_result for diagnostic display |
| 12 | ADR-023 | Workflow Engine design decision record |
| 13 | Sprint 5-6 tests | Desktop + Android test coverage |

## 3. Out of Scope

| # | Item | Rationale |
|---|------|-----------|
| 1 | Conditional steps (if/else) | v0.7 ‚Äî requires expression evaluator |
| 2 | Loops / iteration | v0.7 ‚Äî requires loop semantics |
| 3 | Parallel execution | v0.7 ‚Äî requires concurrency model |
| 4 | Delays / sleeps between steps | v0.7 ‚Äî requires timer infrastructure |
| 5 | Workflow chaining (workflow calls workflow) | v0.7 ‚Äî requires recursive resolution |
| 6 | Triggers (event-driven workflows) | v0.7 ‚Äî requires event bus |
| 7 | Scheduling (time-based execution) | v0.7 ‚Äî requires scheduler daemon |
| 8 | Workflow editing from Mobile | Desktop-authoritative ‚Äî ADR-001 |
| 9 | Workflow versioning / migration | v0.7 ‚Äî premature before real usage |
| 10 | Visual workflow editor | v0.8+ ‚Äî requires GUI framework investment |
| 11 | Undo / rollback of workflow execution | OS actions are fire-and-forget |
| 12 | Retry logic for failed steps | v0.7 ‚Äî needs configurable retry policy |

Workflows live in `Document` ‚Äî same as profiles and context_rules. This
means they flow through the Command/Reducer pipeline (ADR-008) and are
persisted via `JsonRepository`.

### 4.2 Execution Layer Extension

```

  ‚îî‚îÄ‚îÄ execute_action()          ‚Üê atomic action execution

execution.rs (v0.6)
  ‚îú‚îÄ‚îÄ execute_action()          ‚Üê unchanged
  ‚îî‚îÄ‚îÄ execute_workflow()        ‚Üê NEW: sequential step executor
        ‚îú‚îÄ‚îÄ resolves workflow_id via WorkflowRegistry
        ‚îú‚îÄ‚îÄ iterates steps
        ‚îú‚îÄ‚îÄ calls execute_action() per step
        ‚îú‚îÄ‚îÄ stops on first failure
        ‚îî‚îÄ‚îÄ returns WorkflowExecutionResult
```

### 4.3 Protocol Extension

| Direction | Message Type | Purpose |
|-----------|-------------|---------|
| Mobile ‚Üí Desktop | `workflow_invoke` | Request workflow execution |
| Desktop ‚Üí Mobile | `workflow_invoke_result` | Execution result with per-step status |

New messages follow the same schema_version=1 additive pattern as
control_invoke (ADR-022).

### 4.4 Android Extension

- `WorkflowInvokeRequest` ‚Äî outbound serialization
- `SpikeMessageDispatcher` ‚Äî parse workflow_invoke_result
- `WorkflowExecutionResult` ‚Äî diagnostic display model

## 5. Workflow Engine Design

### 5.1 Workflow Model

```rust
pub struct WorkflowId(String);  // timestamp-hex, same as ProfileId

pub struct Workflow {
    pub id: WorkflowId,
    pub name: String,
    pub steps: Vec<WorkflowStep>,
    pub enabled: bool,
}

pub struct WorkflowStep {
    pub action_name: String,     // references ActionRegistry
    pub payload: Option<Value>,  // optional action payload
}
```

### 5.2 WorkflowRegistry

Follows the same pattern as ActionRegistry ‚Äî a HashMap resolving
workflow_id to Workflow reference. Constructed from Document at
execution time.

```rust
pub struct WorkflowRegistry<'a> {
    workflows: HashMap<&'a str, &'a Workflow>,
}

impl<'a> WorkflowRegistry<'a> {
    pub fn new(workflows: &'a [Workflow]) -> Self { ... }
    pub fn resolve(&self, workflow_id: &str) -> Option<&'a Workflow> { ... }
}
```

### 5.3 Execution Model

```rust
pub struct WorkflowExecutionResult {
    pub workflow_id: String,
    pub accepted: bool,           // validation passed
    pub reason: Option<String>,   // rejection reason if !accepted
    pub executed: bool,           // all steps completed
    pub steps: Vec<StepResult>,
    pub execution_error: Option<String>,  // overall error if !executed
}

pub struct StepResult {
    pub step_index: usize,
    pub action_name: String,
    pub executed: bool,
    pub error: Option<String>,
}
```

Execution is sequential. If step N fails:
- Step N's error is recorded
- Steps N+1.. are skipped
- `executed = false`
- `execution_error` contains the failing step's error

### 5.4 Timeout

Each step uses the existing 5s timeout from execution.rs. The overall
workflow timeout is `steps.len() * 5s`. This is a hard ceiling, not a
configurable value.

### 5.5 Workflow Validation (Control Path)

The `workflow_invoke` handler validates:
1. Workflow exists in Document
2. Workflow is enabled
3. All step action_names exist in ActionRegistry

Validation is pure (same principle as validate_button in ADR-022).

## 6. Macro Execution Model (v0.6 Simplification)

v0.6 treats macros as named sequences of atomic actions. This is the
simplest viable model:

- No conditional logic
- No loops
- No parallel execution
- No delays
- No error recovery

Each macro is a `Workflow` with `enabled = true`. The user creates
workflows in the Desktop editor (command-line or future GUI). The
Android client invokes them by workflow_id.

This is deliberately minimal. Real-world usage will reveal whether
conditions, loops, or delays are needed before investing in that
complexity.

## 7. Versioning Strategy

- `schema_version` stays at 1 for all existing message types
- New message types (`workflow_invoke`, `workflow_invoke_result`) start
  at schema_version = 1
- Workflow model is part of Document ‚Äî versioned via the existing
  Document serialization format
- No workflow-specific versioning until the model needs breaking changes

## 9. Migration from v0.5

v0.6 is fully backward-compatible:

- All v0.5 messages unchanged
- `control_invoke` and `control_invoke_result` unchanged
- Document gains a new `workflows` field with default empty Vec
- Backward-compatible deserialization (existing documents load fine)
- No protocol version bump required
- No Android version gate required
