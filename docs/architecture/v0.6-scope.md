# v0.6 — Scope

**Status:** Planning
**Date:** 2026-07-16
**Architecture Review:** 8.8/10 — refinements applied

Normative references: ADR-001, ADR-007, ADR-008, ADR-021, ADR-022,

## 1. Objective

Introduce a Workflow Engine that allows users to define multi-step
automation sequences on the Desktop and execute them from Android.
Workflows compose existing atomic actions into ordered, reusable
sequences. The Workflow Engine is an execution-layer feature, not a
transport feature.

## 4. Architecture Impact

### 4.1 Domain Model Extension

```
Document
  ├── profiles: Vec<Profile>
  ├── context_rules: Vec<ContextRule>
  └── workflows: Vec<Workflow>          ← NEW (v0.6)
        ├── id: WorkflowId
        ├── name: String
        ├── version: WorkflowVersion    ← always 1 in v0.6
        ├── steps: Vec<WorkflowStep>
        │     ├── action_id: ActionId   ← immutable reference
        │     └── payload: Option<Value>
        └── enabled: bool
```

Workflows live in `Document` — same as profiles and context_rules.
This means they flow through the Command/Reducer pipeline (ADR-008)
and are persisted via `JsonRepository`.

### 4.2 Execution Layer Extension (Sprint 2+)

```
execution.rs (v0.5)
  └── execute_action()          ← atomic action execution

execution.rs (v0.6)
  ├── execute_action()          ← unchanged
  └── execute_workflow()        ← NEW: sequential step executor
        ├── resolves workflow_id via WorkflowRegistry
        ├── iterates steps
        ├── calls execute_action() per step
        ├── stops on first failure
        └── returns WorkflowExecutionResult
```

### 4.3 ExecutionTarget Abstraction (Sprint 2)

```rust
pub enum ExecutionTarget {
    Action(ActionId),
    Workflow(WorkflowId),
}
```

The dispatcher resolves the opaque invocation target into an
`ExecutionTarget` and delegates to the appropriate executor. This
keeps a single dispatch path in agent.rs regardless of target type.

### 4.4 Protocol — No New Message Types

v0.6 reuses the existing `control_invoke` / `control_invoke_result`
messages. Android continues to invoke opaque identifiers. The Desktop
resolves whether the identifier represents an Action or a Workflow.

This preserves the opacity principle established in ADR-022. Android
never needs to know that a button triggers a workflow versus a single
action.

The `control_invoke_result` extends with optional per-step fields
for workflow results (additive, schema_version stays at 1).

### 4.5 Android Extension

- `SpikeMessageDispatcher` — parse extended result fields for workflows
- Diagnostic display — show per-step execution status

## 5. Workflow Engine Design

### 5.1 Domain Model

```rust
pub struct WorkflowId(String);  // timestamp-hex, same pattern as ProfileId (ADR-007)

pub struct WorkflowVersion(pub u16);  // initially 1

pub struct Workflow {
    pub id: WorkflowId,
    pub name: String,
    pub version: WorkflowVersion,
    pub steps: Vec<WorkflowStep>,
    pub enabled: bool,
}

pub struct WorkflowStep {
    pub action_id: ActionId,        // immutable reference to ActionRegistry entry
    pub payload: Option<Value>,     // optional action payload
}
```

**Design decisions:**
- `WorkflowStep` references `ActionId`, not `action_name`. Names belong
  in projection/UI; identifiers belong in the domain. Renaming an action
  does not break workflows.
- `WorkflowVersion` is included from the start (always 1) to avoid
  serialization migration when versioning is needed later.
- `payload: Option<Value>` future-proofs parameterized actions.
- Workflows are immutable definitions. Runtime state lives in
  `WorkflowRuntime`.

### 5.2 Runtime Model (Sprint 1 — no execution logic)

```rust
pub struct WorkflowRuntime {
    pub workflow_id: WorkflowId,
    pub state: ExecutionState,
    pub started_at: Option<Instant>,
    pub current_step: Option<usize>,
    pub result: Option<WorkflowExecutionResult>,
}

pub enum ExecutionState {
    Idle,
    Running,
    Completed,
    Failed,
}
```

Runtime objects are created during execution, never serialized, and
never stored in Document. They exist solely for in-memory tracking
during a workflow run.

### 5.3 Protocol — Opaque Invocation

Android sends:
```json
{
  "type": "control_invoke",
  "schema_version": 1,
  "button_id": "opaque-button-id"
}
```

Desktop resolves button_id → ExecutionTarget (Action or Workflow) and
executes accordingly. The response extends with workflow-specific fields
when applicable:

```json
{
  "type": "control_invoke_result",
  "schema_version": 1,
  "button_id": "opaque-button-id",
  "accepted": true,
  "executed": true,
  "workflow_steps": [
    {"step_index": 0, "action": "launch", "executed": true},
    {"step_index": 1, "action": "open_url", "executed": false, "error": "execution_timeout"}
  ]
}
```

`workflow_steps` is absent when the target is a single action. This is
an additive field — existing Android clients ignore it.

### 5.4 Validation

Desktop-side validation expands. All validation is pure (no I/O,
no runtime state):

**Structural validation (Sprint 1):**
- Workflow ID is non-empty
- Workflow name is non-empty
- Workflow has at least one step
- All step action_ids are non-empty

**Referential validation (Sprint 1):**
- All referenced action_ids exist in ActionRegistry
- No duplicate workflow IDs in Document
- WorkflowVersion is supported (currently only 1)

**Runtime validation (Sprint 2):**
- Workflow is enabled
- Workflow exists at invocation time

## 6. Workflow Execution Model (v0.6 Simplification)

v0.6 treats workflows as named sequences of atomic actions. This is
the simplest viable model:

- No conditional logic
- No loops
- No parallel execution
- No delays
- No error recovery

Each workflow is a `Workflow` with `enabled = true`. The user creates
workflows in the Desktop editor (command-line or future GUI). The
Android client invokes them by opaque button_id — the same mechanism
used for single actions.

This is deliberately minimal. Real-world usage will reveal whether
conditions, loops, or delays are needed before investing in that
complexity.

## 7. Versioning Strategy

- `schema_version` stays at 1 for all existing message types
- `control_invoke` and `control_invoke_result` remain unchanged in structure
- `workflow_steps` is an additive field in `control_invoke_result`
- Workflow model is part of Document — versioned via the existing
  Document serialization format
- `WorkflowVersion` field is included in the domain model from the
  start (always 1) to support future migration without serialization
  changes

## 9. Migration from v0.5

v0.6 is fully backward-compatible:

- All v0.5 messages unchanged
- `control_invoke` and `control_invoke_result` unchanged in structure
- Document gains a new `workflows` field with default empty Vec
- Backward-compatible deserialization (existing documents load fine)
- No protocol version bump required
- No Android version gate required
- Android clients ignore unknown fields (`workflow_steps`)
