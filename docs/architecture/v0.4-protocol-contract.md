# v0.4 — Protocol Contract: active_profile_state v1

**Status:** Draft
**Date:** 2026-07-14

Normative references: protocol.md, ADR-020, v0.4-scope.md.

---

## 1. Existing Protocol Baseline

`active_profile_state` is documented in `docs/architecture/protocol.md` (lines
29, 186–251, 253–294). It is listed in the message catalog as a
Desktop → Mobile message and marked "v0.3 — not yet implemented."

The documented draft schema includes:
- `type`: `"active_profile_state"`
- `selection_mode`: `"automatic"` or `"manual"`
- `profile`: nullable object with `id`, `name`, `pages`, `buttons`, `action`

**Status of the pre-v0.4 draft schema:** Planned, documented, not implemented,
not yet wire-stable. The protocol.md header (line 3) marks it as
"Draft v0.1.0-draft" and explicitly warns (line 51):
"All payload examples below are **Draft v0.1.0-draft** and may change until
the wire format is frozen at v1."

v0.4 supersedes the unimplemented draft payload with the first executable,
message-versioned `active_profile_state` contract.

## 2. v1 Schema

```json
{
  "type": "active_profile_state",
  "schema_version": 1,
  "active_profile_id": "abc123..."
}
```

Null variant (no active profile):

```json
{
  "type": "active_profile_state",
  "schema_version": 1,
  "active_profile_id": null
}
```

### 2.1 Fields

| Field | Type | Nullable | Required | Description |
|-------|------|----------|----------|-------------|
| `type` | string | no | yes | Always `"active_profile_state"` |
| `schema_version` | integer | no | yes | `1` for v0.4. Incremented when the payload shape changes. |
| `active_profile_id` | string or null | yes | yes | The currently active Profile ID, or `null` if no profile is active |

### 2.2 Field Justification

| Field | Source | Wire Justification |
|-------|--------|--------------------|
| `type` | protocol.md | Message dispatch discriminator |
| `schema_version` | v0.4 governance | Protocol versioning requirement — allows Android to reject incompatible schema changes |
| `active_profile_id` | `RuntimeProjection` | Core consumer state — Android needs to know which profile is active |

### 2.3 Fields Explicitly Excluded from Wire

| Excluded Field | Source Type | Reason for Exclusion |
|----------------|-------------|---------------------|
| `context_changed` | `RuntimeProjection` | Desktop-internal observer signal. Android has no context observer. Leaking this field would couple Android to desktop observation internals and make the wire protocol brittle. |
| `active_profile_changed` | `RuntimeProjection` | Derived boolean (`previous != active`). If `active_profile_id` changed between messages, Android infers the change from comparison. Exposing the derived flag is redundant. |
| `selection_mode` | `ProfileRuntime` | Transport must not query `ProfileRuntime` (Constraint #5, ADR-014). `RuntimeProjection` does not carry `selection_mode`. Including it would require either transport access to `ProfileRuntime` (rejected) or expanding `RuntimeProjection` (not justified for v0.4). A future version may add it if a concrete consumer requirement emerges. |
| `previous_profile_id` | `RuntimeProjection` | Transition metadata. Under latest-state semantics, a reconnecting Android that never observed P2 would receive `previous = P2` — providing no continuity. No concrete v0.4 Android consumer requirement exists. Excluded to keep the v1 schema as a pure authoritative-state snapshot. |

### 2.4 `profile` Object (protocol.md original draft)

The original protocol.md defines a nested `profile` object with `id`, `name`,
`pages`, `buttons`, `action`. **v0.4 does not include this object.**

Rationale: v0.4 delivers identity-level state changes (which profile is active).
The full Profile definition (pages, buttons, actions) is a separate concern
that may be delivered via request-response or a different message type in a
future version. Including the full profile definition in `active_profile_state`
would make the message too large for a high-frequency state-change signal.

If Android needs the full profile definition in v0.4 (e.g. for the deck UI),
it can request it via an existing mechanism or wait for a future version that
adds a `profile_definition` message.

### 2.5 Nullable Behavior

- `active_profile_id == null`: No profile is currently active. The client
  should display an appropriate empty state.

### 2.6 Message Ordering

After `trusted` or `pair_accepted` response, the desktop sends
`active_profile_state` as the **next message**. This ordering is normative:

```
trusted / pair_accepted
    ↓
active_profile_state  (first — may be a null-profile variant)
    ↓
...future updates...
```

Android must not interpret `active_profile_state` before the trust
acknowledgement.

### 2.7 Latest-State Semantics

- Only the latest `active_profile_state` is meaningful
- Intermediate states may be lost (lossy policy)
- Android must overwrite its retained state on each received message —
  no delta merging, no partial updates
- Idempotent: receiving the same state twice is harmless — Android overwrites
  with identical values

### 2.8 Reconnect Snapshot Semantics

On reconnect, the Android client receives:
1. `trusted` (or pair flow for new devices)
2. An `active_profile_state` message reflecting the current latest state on
   the desktop

This is not a replay — it is a snapshot of the current state delivered after
re-identification. Android must not interpret it as a "new change event."

### 2.9 Unknown Field Behavior

Android ignores unknown fields (current spike uses `JSONObject.optString()`
which returns `""` or `null` for missing keys). This is the correct behavior
for forward compatibility — future schema versions may add fields without
breaking old clients.

### 2.10 Unsupported `schema_version` Behavior

If Android receives a `schema_version` it does not support:
- Log the message with a warning
- Keep the connection alive
- Treat the message as if it was never received (do not update retained state)
- Do NOT attempt to parse fields as version 1
