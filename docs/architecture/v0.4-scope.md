# v0.4 — Projection Transport & Latest-State Delivery

## 1. Objective

Bridge approved `RuntimeProjection` state from the synchronous projection thread
into the existing Tokio-owned WebSocket connection domain using bounded
latest-state semantics, publish a versioned transport DTO only to trusted
Android connections, and prove minimal Android spike consumption of that state.

## 2. Version Title

**Projection Transport Foundation**

## 3. One-Sentence Goal

Deliver approved `RuntimeProjection` state to trusted connected Android clients
via a versioned DTO, using `tokio::sync::watch` for the OS-thread → Tokio
handoff, with no changes to the v0.3 projection pipeline or runtime domain.

## 4. In Scope

### 4.1 ProjectionTransportPublisher

Concrete `ProjectionPublisher` implementation that:
- Holds an `Arc` to the latest-approved-state hub
- Maps `RuntimeProjection` → transport DTO (`ActiveProfileStateMessage`)
- Serializes to JSON once and pushes the serialized message onto the
  `watch` channel
- Rejects unbounded accumulation: never queues historical projections

### 4.2 RuntimeProjection → Transport DTO Mapping

- `RuntimeProjection` remains transport-neutral (no `Serialize` derive)
- A separate `ActiveProfileStateMessage` struct with `serde::Serialize`
  carries only wire-justified fields
- Mapping is a `From<&RuntimeProjection>` or explicit conversion in the
  publisher

### 4.3 `active_profile_state` Schema v1

- `type`: `"active_profile_state"` (per existing protocol.md)
- `schema_version`: 1 (integer, message-local)
- `active_profile_id`: `Option<String>`
- `context_changed`: excluded from wire
- `active_profile_changed`: excluded from wire
- `selection_mode`: excluded from wire (transport must not query `ProfileRuntime`; `RuntimeProjection` does not carry it)
- `previous_profile_id`: excluded from wire (transition metadata, not latest-state snapshot data)

### 4.4 Latest-Approved-State Transport Hub

- `tokio::sync::watch::Sender<Option<Arc<str>>>` where the value is a
  pre-serialized JSON string (immutable, shareable across subscribers)
- Initial value: `None` — no projection has been published yet
- Each `publish()` call replaces: serialize once, `send_replace`
- Connection tasks clone `Receiver` — each sees `changed()`
- New subscribers receive the retained latest value immediately

### 4.5 Trusted Connection Delivery

- Projection delivery begins only after a connection reaches `trusted` state
- `trusted` response precedes the first `active_profile_state` push
- Newly paired devices become eligible at the `pair_accepted` → trust transition
- Untrusted or identified-only connections never receive projection state

### 4.6 Reconnect / Snapshot Delivery

- When a previously-trusted device reconnects:
  1. `identify` → trust lookup succeeds → `trusted` response
  2. Connection task creates a `watch::Receiver`
  3. `Receiver::borrow()` returns the latest retained value immediately
  4. Serialized message is written to the WebSocket sink
- No query into `PublicationPolicy`, `ProjectionEngine`, or `DesktopRuntime`

### 4.7 Minimal Android Spike Consumption

- Existing `spikes/ep-001-discovery-pairing/mobile/` spike:
  - Recognizes `active_profile_state` in `onMessage()` JSON dispatch
  - Reads `schema_version` — rejects unsupported versions
  - Parses approved DTO fields
  - Retains latest state in a simple `var latestProjection: JSONObject?`
  - Exposes diagnostic UI: active profile ID, schema version, last update
- No production architecture (no service, ViewModel, deck rendering)

### 4.8 Shutdown — Lifetime Independence

`watch::Sender` lifetime is independent of the Tokio runtime. During shutdown:

- The server Tokio runtime may terminate before the projection thread exits.
  Connection tasks (and their `watch::Receiver` clones) are aborted when the
  runtime drops — this is delivery loss under the accepted ADR-015 lossy policy.
- The projection thread may briefly continue calling `send_replace()` after
  the runtime has dropped. This is safe because `send_replace()` is an
  infallible, Tokio-runtime-independent operation (Tokio 1.x: "permits sending
  values even when there are no receivers"). Such publications have no consumer
  and are harmless.
- If the projection thread exits first, the `watch::Sender` is dropped and
  any remaining receivers observe a closed channel.
- Either thread may terminate first. No ordering guarantee is required.
- **Join order is non-normative for v0.4.** Main-thread join ordering does not
  establish thread exit ordering or runtime lifetime coordination. No additional
  shutdown acknowledgement, lifecycle barrier, or coordination primitive is
  required.

### 4.9 Executable Transport Integration Tests

Desktop-side integration tests with:
- `MockConnectionSink` replacing real WebSocket
- `ProjectionTransportPublisher` with a `watch` hub
- Serialization round-trip tests
- Multiple receiver tests
- Reconnect / snapshot delivery tests

## 5. Out of Scope

- Production Android app rewrite or architecture
- Android foreground service or background connection management
- Android deck rendering, hot-swapping, or profile switching UI
- Workflows, macros, triggers, or generic event bus
- Protocol-wide universal envelope migration (existing messages unchanged)
- Historical projection replay or retry queues
- Exactly-once delivery guarantees
- Cloud transport, external broker, or pub/sub middleware
- Broad server rewrite or transport ownership refactor
- Fan-out to multiple publisher types (LoggingPublisher + WebSocket)
  — deferred to v0.4.1 if logging evidence is needed alongside transport
- Write timeouts, circuit breakers, or back-pressure signalling
- Async runtime migration for the projection thread
- Android reconnection logic beyond the existing `autoReconnect()` spike

## 6. Normative Constraints (inherited from v0.3)

| # | Constraint | Source |
|---|-----------|--------|
| 1 | DesktopRuntime remains authoritative | ADR-014 |
| 2 | project() remains pure, deterministic, side-effect free | ADR-014 |
| 3 | PublicationPolicy exclusively owns suppression | ADR-015 |
| 4 | ProjectionPublisher is the transport boundary | ADR-014, transport-contract |
| 5 | Publisher must not access DesktopRuntime or ProfileRuntime | ADR-014 |
| 6 | Publisher must not filter or suppress projections | ADR-015 |
| 7 | Latest-value semantics are lossy — newest value wins | ADR-015 |
| 8 | No unbounded historical queue | ADR-015, ADR-016 |
| 9 | Observer never waits on transport I/O | ADR-016 |
| 10 | Projection thread never acquires DesktopRuntime lock | ADR-016 |
| 11 | Projection thread must not perform WebSocket I/O | v0.4 architecture decision |
| 12 | Protocol changes must be versioned | v0.4 governance |
| 13 | Reconnect semantics must be explicit | ADR-015 |
| 14 | Shutdown ownership must be explicit | ADR-016 |
