# v0.4 ‚Äî Projection Transport

**Status:** Pre-release ‚Äî pending final release audit and tag decision
**Date:** 2026-07-14
**Version:** amd-desktop 0.4.0

## Summary

The projection transport bridges approved `RuntimeProjection` state from the
synchronous projection thread into the Tokio-owned WebSocket connection domain
using bounded latest-state semantics. Trusted Android clients receive versioned
projection state via a dedicated push mechanism. The Android discovery/pairing
spike validates strict v1 projection reception.

Architectural outcome:

```
Desktop active-profile authority
       ‚Üì
RuntimeProjection
       ‚Üì
retained projection transport state (watch channel)
       ‚Üì
trusted WebSocket delivery
       ‚Üì
Android strict projection receiver
```

## What's new

### Projection Transport (ADR-020)

- **`ProjectionTransportPublisher`** ‚Äî concrete `ProjectionPublisher` that maps
  `RuntimeProjection` ‚Üí `ActiveProfileStateMessage` DTO, serializes to JSON,
  and pushes onto `tokio::sync::watch`.
- **`watch::Sender<Option<Arc<str>>>`** ‚Äî OS-thread-safe latest-state handoff
  from projection thread to Tokio runtime. No unbounded accumulation, no
  per-client queues.
- **Trusted delivery only** ‚Äî projection state is delivered exclusively to
  connections that have reached `trusted` or `pair_accepted`.
- **Latest-state semantics** ‚Äî reconnecting clients receive the retained
  snapshot immediately. Slow consumers skip intermediate values.

### Wire contract (v1)

```json
{
  "type": "active_profile_state",
  "schema_version": 1,
  "active_profile_id": "coding"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `type` | string | yes | Always `"active_profile_state"` |
| `schema_version` | integer | yes | The exact integer `1` for v1 |
| `active_profile_id` | string or null | yes | Active Profile ID or explicit `null` |

### Android strict projection receiver

- `ActiveProfileStateMessage` parser with strict v1 validation
- Rejects unsupported or invalid `schema_version` (non-integer values,
  fractional `1.0`, `1.5`, out-of-range)
- Retains latest accepted projection in Activity memory
- 18 JUnit unit tests

## Preserved invariants

- Desktop remains the exclusive active-profile and context-resolution authority
- Android does not resolve context, persist projection state, or select profiles
- Untrusted clients never receive projection state
- No cloud dependency introduced
- v0.4 does not transport full Profile definitions (pages, buttons, actions)

### Test infrastructure note

Local JVM unit tests use `org.json:json:20231013` as a `testImplementation`
dependency because Android SDK `org.json` stubs throw `"Method ... not mocked"`
during local JVM unit testing. This is test infrastructure only ‚Äî it is not
part of the wire protocol.

### Known non-blocking validation warning

Android Gradle validation emitted an SDK XML version compatibility warning:

"SDK processing. This version only understands SDK XML versions up to 3 but an
SDK XML file of version 4 was encountered."

The warning did not block unit tests, `assembleDebug`, or `lint`; all completed
successfully. It is recorded as validation-environment/tooling debt and is not
classified as a v0.4 projection transport defect.

## Known limitations

- **Spike-only Android** ‚Äî No production Android client exists.
  The discovery/pairing spike is disposable.
- **Memory-only projection** ‚Äî Android does not persist projection state.
  State is lost on process death.
- **No full profile transport** ‚Äî v0.4 delivers only the active Profile ID.
  Full Profile definitions (pages, buttons, actions) are deferred.

## Deferred to v0.4.1 / v0.5

- Full Profile definition transport (pages, buttons, actions)
- Android production app architecture
- Android projection persistence
- Desktop-side fan-out to multiple publisher types
