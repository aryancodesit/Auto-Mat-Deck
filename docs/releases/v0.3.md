# v0.3 — Projection Pipeline

**Release date:** 2026-07-13
**Version:** md-desktop 0.3.0

## Summary

The projection pipeline converts raw context observations into filtered, publisher-ready projections. This completes the data path: **Win32 window query → Observer → RuntimeTransition → ProjectionEngine → PublicationPolicy → ProjectionPublisher**. Thread ownership is codified via ADR-016 with Condvar-based notification between observer and projection threads.

## What's new

### Projection Pipeline (ADR-014, ADR-015, ADR-016)

- **TransitionCell** — bounded O(1) latest-value synchronization cell with Condvar notification (wait_and_take_timeout). Observer overwrites without blocking; projection thread wakes on notify.
- **ProjectionEngine** — pure function RuntimeTransition → RuntimeProjection. Maps context changes and active profile changes into a deterministic projection.
- **RuntimeProjection** — struct with context_changed, ctive_profile_changed, previous_profile_id, ctive_profile_id fields. Zero-copy field references to shared model data.
- **PublicationPolicy** — rate-limits and deduplicates projections before publication. should_publish() returns alse for consecutive identical projections.
- **ProjectionPublisher** — trait with publish(&self, projection: &RuntimeProjection). LoggingPublisher is the bootstrap implementation.
- **Projection thread** — dedicated thread per ADR-016. Receives transitions via TransitionCell, runs ProjectionEngine then PublicationPolicy, publishes to ProjectionPublisher. Shutdown via watch channel; joins before observer thread.

## Concurrency model

- **Observer thread** (Win32 polling) → stores RuntimeTransition into TransitionCell, calls 
otify_one.
- **Projection thread** → wait_and_take_timeout on TransitionCell, wakes on notify or 200ms shutdown check.
- **Main thread** → owns GUI, tray, server. Drops shutdown_tx to close watch channel.
- **Shutdown ordering**: signal → projection joins → observer joins (per ADR-016).

## Known limitations

- **LoggingPublisher only** — projections are logged to console. No transport (WebSocket, serial) exists yet.
- **50–200ms shutdown latency** — projection thread re-checks shutdown at 200ms intervals while idle (Condvar wait_timeout fallback).
- **No multi-consumer support** — single projection thread, single publisher.
- **No persistence** — projections are ephemeral; no replay or state recovery on restart.

## Deferred to v0.4

- WebSocket transport (WebSocketPublisher)
- JSON serialization for wire format
- Android deck client connection and reconnection
- Fan-out to multiple consumers
- Async runtime migration (if needed)

## Architecture freeze

v0.3 is **frozen**. No feature additions after this point. Transport and connectivity belong in v0.4 and build on top of the projection pipeline without changing it.
