# ADR-008: Domain mutations use a Command/Reducer pattern

**Status:** Implemented
**Date:** 2026-07-12
**Applies To:** v0.3+
**Supersedes:** —
**Superseded By:** —
**Engineering Phase:** EP-004
**Release:** v0.3

## Context

Domain mutations for profiles, pages, and buttons were growing ad-hoc.
Without a uniform mutation path, every new operation required:

- A new `AppState` method.
- Manual boilerplate for ID validation and invariant enforcement.
- Duplicated error handling.
- No support for undo/redo, audit, or sync in the future.

Device mutations are still handled by direct `AppState` methods. Those may
migrate to commands if undo/redo is needed for device operations.

## Decision

All profile/page/button mutations pass through a pure reducer:

```
Command → apply(Document) → Result<Document, CommandError>
```

The reducer is defined in `src/command.rs` as a module with no dependencies
on GUI types, WebSocket types, or persistence.

### Key choices

1. **Identity is supplied by the caller, not generated by the reducer.**
   Commands carry entity IDs for every persistent entity they create
   (`profile_id`, `initial_page_id` for CreateProfile; `page_id` for AddPage;
   the entire `Button` struct for AddButton). The caller creates these via
   `ProfileId::new()` etc. The reducer never calls ID generation functions
   and uses no magic/constant ID strings. Identical inputs always produce
   identical outputs — the reducer is referentially transparent.
2. **Clone-and-mutate, not mutation in-place.** The reducer clones the
   Document, applies the change, and returns the result. This makes it safe
   to test without mocking, and leaves room for undo snapshots.
3. **Typed errors, not panics or strings.** Every failure mode is a named
   `CommandError` variant.
4. **No persistence in the reducer.** The caller decides when to persist.
   This keeps the reducer testable without disk I/O and allows batching
   multiple commands before a single save.
5. **AppState::dispatch as integration point.** The GUI calls one method
   which delegates to the reducer and replaces the document on success.

### Error precedence

Deletion commands validate entity existence before invariant checks:

```
DeleteProfile { profile_id }:
    1. Resolve profile_id → NotFound if missing
    2. Check is last profile → CannotDeleteLastProfile if true
    3. Delete

DeletePage { profile_id, page_id }:
    1. Resolve profile_id → ProfileNotFound if missing
    2. Resolve page_id → PageNotFound if missing
    3. Check is last page → CannotDeleteLastPage if true
    4. Delete
```

### What was deferred

- Undo/redo (needs inverse commands or snapshot stack).
- Audit logging (needs command serialization).
- Network sync (needs protocol integration).
- Device command migration (not needed yet).

## Consequences

- **Positive:** New operations follow a predictable pattern: add a variant to
  `Command`, add a branch in `apply()`. Testing is pure function testing.
  Invariants are checked in one place.
- **Positive:** The reducer is deterministic with respect to its inputs —
  no I/O, no time-based ID generation, no external state. Identical inputs
  always produce identical outputs.
- **Positive:** All profile/page/button operations were implemented with
  22 unit tests, with no GUI or persistence mocking.
- **Negative:** Every mutation clones the Document. For the current data size
  (~dozen devices, ~handful of profiles) this is negligible. If documents
  grow to thousands of entries, a diff-based approach may be needed.
- **Neutral:** Device mutations live outside the command system for now.
  This is intentional — they have different invariants and no editor UI.

## Rationale

The pattern is well-established (see: Redux, Elm architecture, Event
Sourcing). It defers the complex features (undo, sync, audit) without
requiring a rewrite when they arrive — they all compose over the same
`apply()` function. The identity-supplied-by-caller design was adopted in
Sprint 3A after audit found that internal ID generation made the reducer
non-deterministic and would break command replay.
